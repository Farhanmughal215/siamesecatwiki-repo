# siamese.wiki — Developer Handoff

Static site. No build step, no framework. Vanilla HTML + CSS + JS, markdown content loaded via fetch, procedurally-generated SVG placeholders until real images land.

---

## Quick start

```bash
# Any static file server works. From the project root:
python3 -m http.server 8000
# then open http://localhost:8000/site/
```

Or use Vite / Live Server / serve / whatever you prefer. There's no build pipeline to run.

## How it works

### Article rendering
1. Each article lives as a markdown file in `/articles/`.
2. `/articles/manifest.json` indexes them all (generated from the markdown — see below).
3. `site/article.html?slug=seal-point-siamese-the-complete-guide` fetches the matching markdown and renders it.
4. Markdown renderer is hand-rolled in `site/site.js` (`renderMarkdown()`). Handles headings, tables, lists, blockquotes, footnotes (`[1]` → `<sup>`), and CTA blocks (`:::cta ... :::`).

### Image placeholders
`site/thumbs.js` generates a deterministic SVG placeholder per article — category-colored, with a big watermark number. When a real image exists at the expected path, it replaces the placeholder (via `<img onerror>` fallback).

### Café CTA tiers
Each article has a `ctaTier` in the manifest:
- **`loud`** — full sponsor footer block + sidebar mini-CTA (personality, colors, behavior topics)
- **`medium`** — full sponsor footer block (general reference)
- **`subtle`** — single-line disclosure only (health topics; no inline CTAs ever)

---

## Regenerating the manifest

The manifest is generated by scanning the markdown files. If you add/edit/remove articles, regenerate with a Node script — or port my Python equivalent. Here's the logic (currently baked into `site/site.js` via the snapshot in `articles/manifest.json`):

For each `articles/NN-slug.md`:
```
{
  "num": <N>,                     // from filename
  "slug": "<slug>",
  "file": "NN-slug.md",
  "title": <first H1>,
  "deck": <first italic line under H1>,
  "quickAnswer": <first blockquote>,
  "category": <mapped from slug keywords>,  // see site/site.js `categoryFor()`
  "ctaTier": <loud|medium|subtle>,          // see site/site.js `tierFor()`
  "words": <word count>,
  "readingMins": <ceil(words/220)>
}
```

Recommendation: **port to a real SSG (Astro, Eleventy, or Next.js)** when you have >30 articles. The current setup is intentionally minimal — no build step means editors can see changes instantly — but it won't scale.

## Deploying

Any static host works: Netlify, Vercel, Cloudflare Pages, GitHub Pages, S3 + CloudFront.

Point the root at `/site/`, not the project root. The articles/ and assets/ folders need to be accessible at `../articles/` and `../assets/` from the site pages (relative paths in the current code). Easiest deploy: upload the whole project as-is, set the public path to `/site/`.

- Which article(s) use it

Skip this step and the wiki fails its own editorial standards. Wikimedia images in particular require attribution.
